筛选目标

这批页面的目标不是随机展示样本，而是筛出最能说明“主 agent 协调多个子 agent 完成复杂搜索任务”这一方案合理性的轨迹。筛选时优先考虑主子 agent 分工是否清楚、最终答案是否正确、证据链是否被主 agent 汇总和修复，而不是单纯追求轨迹最长或子任务最多。

主子轨迹匹配规则

每个子 agent 页面都必须通过精确的 (question, goal) 二元组匹配到主轨迹。也就是说，子轨迹里的 question 必须和主 agent 的问题完全一致，子轨迹里的 goal 必须来自主轨迹中实际发出的 call_sub_agent prompt 列表。只按 question 匹配是不够的，因为同一个问题下可能有很多不同子任务；只按 goal 匹配也不够安全，因为不同问题可能出现相似目标。

生成时会做强校验：如果某个实际 goal 找不到对应子轨迹，或者同一个 (question, goal) 匹配到多条子轨迹，脚本会直接失败，不会生成可能错配的页面。

硬性过滤条件

  • 所有样本都必须通过精确的 (question, goal) 匹配，不能错配子 agent 轨迹。
  • 最终答案必须和参考答案一致；如果是缩写、大小写、标点、别名或括号补充信息差异，必须能明确说明语义等价。
  • 前两档主 agent 必须正常结束，即 termination = answer，并且题面关键约束必须能在主 agent 最终解释或某个子 agent 报告中找到对应验证。
  • 第三档是专门展示强制回答机制的例外，要求 termination = generate an answer as token limit reached (rolled back) 且最终答案正确；这类样本允许存在少量未完全闭合的约束，但必须在页面说明中明确标注，不能作为“完美证据闭合”样本使用。
  • 轨迹必须包含有意义的子 agent 分工；如果过短且过简单、看不出协作价值，则不作为核心展示。
  • 前两档即使答案正确，如果主要过程依赖大量错误候选、反复否定、多个子 agent 失败或大面积 token_limit，也不进入展示集；第三档会保留这些状态用于说明 forced-answer 边界行为。

样本分档

核心干净样本

这是最严格的一档：主 agent 正常结束、答案正确、所有实际调用到的子 agent 都是 status = completed，并且子任务拆分足够清楚，能直接看出主子 agent 协作的价值。

强协作样本（含子 agent 瑕疵）

这一档的主 agent 仍然必须正常结束且答案正确，但允许少量子 agent 出现 token_limit 或无可用内容/error。这里的“瑕疵”只指子 agent 的执行状态或局部检索不完整，不代表题面约束可以缺失。只有当主 agent 通过后续子任务、直接访问来源、计算校验或显式交叉验证把这些瑕疵补上，并且所有关键约束都能在轨迹中找到对应证据时，样本才会被保留。注意：这一档也不允许主 agent 超长回滚。

主 agent 超长强制回答样本

这一档用于展示系统在主 agent 触发上下文或轮次限制后，如何插入强制收尾指令，要求模型停止工具调用并输出 <explanation> 与 <answer>。这一档的最终答案必须正确，但不要求达到前两档的“完美证据闭合”标准；页面中会保留并展示子 agent 的 error、token_limit 或 max_calls 瑕疵，读者可以看到强制回答机制是在什么样的长轨迹尾部发生的。
